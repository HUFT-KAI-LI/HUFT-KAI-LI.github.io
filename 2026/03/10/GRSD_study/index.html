GRID学习记录

发表于2026-03-10|更新于2026-03-10|深度学习、tansfomer、生成式推荐
|总字数:862|阅读时长:2分钟|浏览量:

解决的问题：

1.冷启动的问题（cold start）：传统的推荐系统，没有点击此时很难学习embedding，但是这个传统的双塔模型可以在一定的程度上解决这个问题，但是生成式推荐系统能够直接根据商品或者就是帖子生成embedding，不需要交互行为。
2.超大规模的检索问题，传统的推荐的ANN search，需要ANN index，以及vector database，但是生成式的推荐仅仅需要Transform直接生成即可
3.推荐模型具有语义的理解能力，同时能够做到更多的任务，比如推荐+检索、推荐+对话等等新的尝试
4.建模更加复杂的用户行为，主要是大模型具有更加建模能力

存在的问题：
1.生成错误的序列ID
2.计算成本会更加的高
3.推理速度会更加的慢
4.设计会更加的复杂等

1. encoder层面→embedding

1.1 ymal文档当中的参数

1.1.1 项目的整体 pipeline 是什么？

针对清洗之后的数据，item text → LLM encoder(T5) → item embedding(2048，离散) → Residual quantization(量化token) → Semantic ID(离散序列向量数据) → Transformer (TIGER) → 生成推荐

1.1.2 为什么用 LLM 生成 embedding？

item数据有语义文本，LLM又能够捕捉到语义信息，同时embedding能够展示物品的相似度

1.1.3 为什么使用 T5Encoder？

开源，能够处理语义信息，而且我们仅仅需要encoder不需要decoder

1.1.4 论文代码当中的embedding 是怎么得到的？

对于T5的token embedding输出做了平均的池化，此问题可以在1.1.5得到详细的解答

1.1.5 item text如何得到item embedding？

item text(商品内容) → tokenizer(分词器) → T5 encoder(编码器) → token embeddings(sequence embedding) → mean pooling → item embedding(维度是2048),之后的操作就是量化得到语义ID

针对Transform，其输出的是sequence embedding，也就是序列向量，但是针对推荐系统需要的是item embedding，因此需要将sequence embedding转换成item embedding，此时使用的就是mean pooling

1.1.6 项目使用Lightning进行管理？

对于大规模的训练而言还是PyTorch Lightning比较好,PyTorch Lightning 是 PyTorch 的高层封装，它保留了 PyTorch 的灵活性，同时自动管理训练循环、日志、checkpoint 和多GPU训练等工程细节，从而减少大量 boilerplate code，使模型开发更加专注于核心逻辑。

1.1.7 tokenizr的大小是128，为什么是128，超出了如何操作？

transform需要固定长度的输入，所以需要设定max_length，当一句话转化为token大于128，此时我们进行截断，小于直接进行padding补充0的操作；一般而言针对推荐系统而言token128已经是足够大的

10.其他的小项

10.1 为什么 GRID 不直接用 LLM embedding 做推荐，而要再做 Residual Quantization？

Transformer 生成的是离散 token，但是LLM embedding是连续的，所以需要量化；连续的2048的空间很大，此时我们使用quantization 中，会训练一个codebook，一个离散的词表大小为255(向量字典)，此时就减少了大小,此时的向量索引给到新的token，比如token = 37，意味此时这个就离这个最近，当然这个codebook的token可以有多层；支持自回归生成推荐，他是tiger
